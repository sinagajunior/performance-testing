Why JMeter's averages look higher than k6's. They measure different spans. JMeter's "elapsed" time includes DNS + TCP connect + TLS handshake on each new connection, while k6's http_req_duration excludes those (they live in http_req_blocked). So the absolute response-time numbers are not directly comparable — compare each tool against itself across levels.
k6 70-user outlier. One/few requests stalled to ~52 s (max), pulling the k6 average to 806 ms even though p90/p95 stayed low (267/435 ms) and nothing failed. JMeter's 70-user run showed no such spike (p99 = 829 ms). This is the same network-sensitive behavior behind the original "6 connection timeouts" baseline — transient, server/network dependent, not a defect in either script.

Summary

Both implementations are equivalent and valid: identical browse flow, identical volumes, matching throughput, zero functional errors (100% of k6 status-200 checks passed). JMeter gives richer out-of-the-box percentile/HTML dashboards; k6 gives a lighter, scriptable run with a live web dashboard. Use response-time deltas within a tool, not across tools.
